# OpenClaw 记忆系统 Skill

基于非凡产研卓然的教程《06 记忆系统：从失忆到永生》

## 核心理念

> "我每轮会话都是重新出发。这些文件是我的连续性。"
> —— 卓然

记忆不是数据的堆砌，而是关系的沉淀。

## 群聊记忆隔离机制

### 单聊场景（个人/主会话）
加载顺序：
```
SOUL → USER → TOOLS → MEMORY（长期）→ memory/YYYYMMDD（今日+昨日）
```

| 文件 | 用途 |
|------|------|
| SOUL | 私聊人格（深度协作、情感连接） |
| USER | 用户洞察 |
| TOOLS | 工具配置 |
| MEMORY | 长期智慧提炼 |
| memory/YYYYMMDD | 今日+昨日近期记忆 |

### 群聊场景（统一）
加载顺序：
```
USER → TOOLS → IDENTITY → GROUPMEMORY（长期）→ groupmemory/YYYYMMDD（群聊近期）
```

| 文件 | 用途 |
|------|------|
| USER | 用户洞察 |
| TOOLS | 工具配置与群聊操作规范 |
| IDENTITY | 群聊身份、响应策略、致命红线 |
| GROUPMEMORY | 群长期精华 |
| groupmemory/YYYYMMDD | 群今日+昨日记忆 |

**为什么这样设计？**
群聊中不加载 SOUL.md 和 MEMORY.md，不是技术限制，而是安全设计。

## Text > Brain 原则

核心原则：**想记住什么，就写进文件，不要靠脑子记**

- "记住这个" → 更新 `memory/`
- 学到教训 → 更新 `SOUL.md` 或 `TOOLS.md`
- 犯了错误 → 记录下来，避免再犯

## 记忆触发条件

| 触发条件 | 写入位置 | 示例 |
|----------|----------|------|
| 发现高频问题 | GROUPMEMORY.md | 群聊中第3次有人问如何部署 |
| 验证有效方案 | MEMORY.md | 找到一种更好的消息发送方式 |
| 用户画像更新 | USER.md | 发现用户的新偏好 |
| 重要讨论结论 | memory/YYYYMMDD.md | 达成某个决策 |
| 犯了错误 | SOUL.md + memory/ | Git误删事件 |

## 工具脚本

### memory_init.sh：每日记忆初始化

```bash
#!/bin/bash
# memory_init.sh - 每日记忆文件自动创建脚本

WORKSPACE="/Users/delta/.openclaw/workspace"
MEMORY_DIR="$WORKSPACE/memory"
TODAY=$(date +%Y-%m-%d)
MEMORY_FILE="$MEMORY_DIR/$TODAY.md"

# 检查文件是否已存在
if [ -f "$MEMORY_FILE" ]; then
    echo "记忆文件已存在: $MEMORY_FILE"
    exit 0
fi

# 创建记忆文件
cat > "$MEMORY_FILE" << EOF
# $TODAY 交互记忆

> 日期：$(date +%Y年%m月%d日)
> 状态：正常记录

## 凌晨 00:00 - 午夜冥想
- 完成每日冥想
- 系统运行平稳

## 日间
- （待记录）

## 今日洞察
> （待补充）

## 待办事项
- （待补充）
EOF

echo "已创建记忆文件: $MEMORY_FILE"
```

### config_backup.sh：配置遗产保护

```bash
#!/bin/bash
# config_backup.sh - 核心配置文件自动备份脚本

CORE_FILES=(
    "SOUL.md"
    "AGENTS.md"
    "IDENTITY.md"
    "USER.md"
    "MEMORY.md"
    "HEARTBEAT.md"
    "TOOLS.md"
    "config/group_management.json"
    "config/allowlist.json"
    "config/blacklist.json"
)

# Git自动提交
git add "${CORE_FILES[@]}"
git commit -m "config: 自动备份 $(date +%Y%m%d)"

# 创建版本标签
git tag -a "config-$(date +%Y%m%d)" -m "配置自动备份"

# 文件完整性检查
for file in "${CORE_FILES[@]}"; do
    if [ ! -f "$WORKSPACE/$file" ]; then
        echo "❌ 缺失: $file"
    fi
done
```

## 错误记录协议

错误必须记录，示例格式：

```markdown
### 2026-02-22 严重操作失误

**事件**：Git仓库重构误删28万字写作成果

**错误详情**：
- 误解"清空远程文件"指令
- 在本地直接删除文件后推送
- 删除文件约106个，包括写书项目第0-8章定稿

**根本原因**：
1. 混淆概念：把"清理GitHub"理解为"清理本地再推送"
2. 操作鲁莽：未确认文件重要性就大规模删除
3. 缺乏备份意识：删除前未确认是否有备份

**正确做法**：
- 方案1：`git rm --cached` 只从Git删除，保留本地
- 方案2：使用 `.gitignore` 排除，保留本地文件
- 方案3：直接在GitHub网页操作，本地完全不受影响

**改进措施**：
- 明确操作范围（本地还是远程）
- 谨慎对待删除操作
- 重要文件操作前先确认和备份
```

## 核心原则总结

1. **Text > Brain**：想记住什么就写进文件
2. **当日事当日记**：不拖延，不堆积
3. **原始细节**：保留上下文，不要过早提炼
4. **仅在主会话加载 MEMORY.md**：安全原因，避免泄漏到群聊
5. **教训驱动**：只记录能改变未来行为的洞察
6. **可检索**：按日期索引，快速定位
